# 并发测试中的失败报告

## 问题

如何理解书中这句话：

```text
在 java.util.concurrent 的一致性测试中，要把各种故障与特定测试明确关联起来；
JSR 166 的测试基类会在 tearDown 阶段传递和报告失败信息；
每个测试必须等待它创建的全部线程结束后才能完成。
```

在并发测试中，不能只关注主测试线程有没有失败。
测试期间创建的工作线程如果抛出异常或断言失败，也必须被测试框架捕获并报告。

## 为什么普通测试方式不够

普通单线程测试里，如果断言失败：

```java
assertTrue(false);
```

测试框架马上就能知道这个测试失败。

但并发测试经常会启动工作线程：

```java
new Thread(new Runnable() {
    @Override
    public void run() {
        throw new RuntimeException("worker failed");
    }
}).start();
```

如果异常发生在工作线程中，主测试线程未必能感知到。

测试方法可能已经正常返回，测试框架就会认为测试通过。
但实际上，测试创建的工作线程已经失败了。

这会导致“假通过”。

## 需要把失败和具体测试关联起来

书中说：

```text
一定要将各种故障与特定的测试明确地关联起来。
```

意思是：

```text
哪个测试创建的线程失败了；
失败线程抛出了什么异常；
这个异常应该让对应测试失败；
失败信息应该能帮助定位问题。
```

如果不做这个关联，就可能出现：

```text
线程 A 是 testOne 创建的；
它在 testOne 结束后才失败；
测试框架却不知道这个失败属于 testOne。
```

更糟糕的是，这个后台线程可能继续影响后面的测试。

## 并发测试通常要做的事

一个可靠的并发测试框架通常会做这些事情：

```text
捕获工作线程中的异常；
把异常保存到测试对象中；
等待测试创建的所有线程结束；
在测试结束阶段统一检查是否有子线程失败；
如果有失败，把失败重新报告给测试框架。
```

也就是说，测试创建的线程不能没人管。

测试必须负责：

```text
启动线程；
等待线程结束；
收集线程异常；
清理线程资源。
```

## 为什么要等待所有线程结束

并发测试结束时，必须等待它创建的所有线程结束。

原因有两个：

```text
如果线程之后才失败，测试框架可能漏报失败；
如果线程继续运行，它可能污染后续测试。
```

例如：

```text
testA 创建了后台线程；
testA 方法返回；
testB 开始执行；
testA 的后台线程还在修改共享状态；
testB 莫名其妙失败。
```

这类问题很难诊断。

所以并发测试里有一个基本原则：

```text
谁创建线程，谁负责回收线程。
```

## 总结

并发测试要避免“假通过”和“测试污染”。

关键要求是：

```text
工作线程中的失败必须传回主测试结果；
测试创建的所有线程必须在测试结束前被回收；
失败信息必须能定位到具体测试。
```
