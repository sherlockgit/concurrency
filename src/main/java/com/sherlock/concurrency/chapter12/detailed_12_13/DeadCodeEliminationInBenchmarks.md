# 基准测试中的死代码消除

## 问题

如何理解书中这句话：

> 在编写优秀的基准测试程序时，一个需要面对的挑战就是：优化编译器能找出并删除那些不会对输出结果产生任何影响的无用代码。基准测试通常不会执行任何计算，因此它们很容易在编译器的优化过程中被消除。在大多数情况下，编译器从程序中删除无用代码都是一种优化措施，但对于基准测试程序来说却是一个大问题，因为这将使得被测试的内容变得更少。如果幸运的话，编译器将删除整个程序中的无用代码，从而得到一些明显虚假的测试数据。但如果不幸运的话，编译器在消除无用代码后将提高程序的执行速度，从而使你的测试结果出错。

## 核心意思

这段话在讲：

**基准测试代码很容易被编译器优化掉，导致你以为测了某段代码，实际上那段代码根本没有真正执行。**

如果一段计算的结果没有被使用，编译器可能认为：

```text
这段代码对程序最终输出没有任何影响
```

于是它会直接删除这段代码。

这种优化叫：

```text
死代码消除
Dead Code Elimination
```

## 什么是死代码

死代码就是：

```text
执行了也不会影响程序最终结果的代码
```

例如：

```java
Math.sqrt(100);
```

这行代码虽然做了计算，但结果没有保存、没有返回、没有打印，也没有影响任何状态。

所以从程序最终行为来看，它等于没做。

编译器可能会认为它是无用代码。

## 一个错误的基准测试例子

假设你想测试 `Math.sqrt` 的性能：

```java
long start = System.nanoTime();

for (int i = 0; i < 1_000_000; i++) {
    Math.sqrt(i);
}

long end = System.nanoTime();
System.out.println(end - start);
```

表面上看，这段代码执行了 100 万次 `Math.sqrt`。

但问题是：

```java
Math.sqrt(i);
```

计算结果没有被使用。

编译器可能判断：

```text
这个计算不会影响最终输出，可以删除。
```

最终可能优化成：

```java
long start = System.nanoTime();

for (int i = 0; i < 1_000_000; i++) {
    // 什么也不做
}

long end = System.nanoTime();
System.out.println(end - start);
```

甚至整个循环也可能被删除。

这时你测到的就不是 `Math.sqrt` 的耗时，而是一个被优化后的空测试。

## 为什么基准测试特别容易被消除

真实业务代码通常会产生真实效果：

```text
写数据库
返回响应
修改对象状态
发送网络请求
更新缓存
输出结果
```

这些效果会影响程序后续行为。

但基准测试经常只是为了“测一下这段代码多快”。

例如：

```text
循环做很多次计算
但不关心计算结果
只关心耗时
```

这在编译器看来就很危险：

```text
你既然不用计算结果，那我就可以把计算删掉。
```

所以书里说：

**基准测试程序很容易在编译器优化过程中被消除。**

## 为什么这在普通程序中是好事

在普通程序中，删除无用代码通常是好优化。

例如：

```java
int x = expensiveComputation();
return 1;
```

如果 `x` 没有被使用，并且 `expensiveComputation()` 没有副作用，那么编译器删除它是合理的。

这样程序会更快。

所以死代码消除本身不是坏事。

它的问题在于：

```text
基准测试的目的就是测这段代码的成本。
```

如果编译器把它删掉，测试就失去了意义。

## “幸运”和“不幸运”是什么意思

书里说：

```text
如果幸运的话，编译器将删除整个程序中的无用代码，
从而得到一些明显虚假的测试数据。
```

例如你测出：

```text
执行 1 亿次计算只用了 1 纳秒
```

这个结果明显不可能。

你一眼就能发现测试坏了。

这反而是“幸运”的，因为错误非常明显。

不幸运的情况是：

```text
编译器只删除了一部分被测代码
或者只优化掉了某些关键路径
```

这时结果可能看起来还算合理：

```text
真实耗时应该是 100ms
测试结果显示 35ms
```

这个数字不一定离谱，所以你可能误以为测试是正确的。

这才更危险。

## 如何避免被死代码消除

基本原则是：

```text
让被测代码的结果被使用
```

例如：

```java
double sum = 0;

long start = System.nanoTime();

for (int i = 0; i < 1_000_000; i++) {
    sum += Math.sqrt(i);
}

long end = System.nanoTime();

System.out.println(sum);
System.out.println(end - start);
```

这里 `Math.sqrt(i)` 的结果会影响 `sum`。

而 `sum` 又被打印出来。

因此编译器不能轻易把 `Math.sqrt(i)` 删除。

## 仍然要小心其他优化

即使结果被使用，也不代表测试一定正确。

编译器还可能做其他优化：

```text
常量折叠
循环展开
方法内联
逃逸分析
锁消除
分支预测优化
```

例如：

```java
for (int i = 0; i < 1_000_000; i++) {
    sum += Math.sqrt(100);
}
```

因为 `Math.sqrt(100)` 每次结果都一样，编译器可能提前算出结果。

所以基准测试还要避免输入过于固定、路径过于单一。

## JMH 和 Blackhole

在 Java 中，更推荐使用 JMH 做微基准测试。

JMH 是专门为 JVM 微基准测试设计的工具。

它会处理很多容易踩坑的问题，例如：

```text
预热
多轮测量
防止死代码消除
避免常量折叠
控制 JVM fork
统计结果
```

JMH 中常用 `Blackhole` 消费结果：

```java
@Benchmark
public void test(Blackhole blackhole) {
    blackhole.consume(Math.sqrt(123));
}
```

`Blackhole` 的作用是告诉编译器：

```text
这个计算结果被使用了，不能简单删掉。
```

但它又不会把输出打印到控制台，避免 I/O 干扰测试。

## 和本章测试的关系

第 12 章前面的 `TimedPutTakeTest` 不只是空循环。

它会：

```text
生产者生成数据
put 到缓冲区
消费者从缓冲区 take
累加 putSum 和 takeSum
最后断言 putSum == takeSum
```

因为结果会参与校验：

```java
assertEquals(putSum.get(), takeSum.get(), "putSum should equal takeSum");
```

所以这些 `put` 和 `take` 操作不能被简单视为无用代码。

这也是为什么书中要使用校验和。

它不仅验证正确性，也防止测试内容被优化器当成无意义计算删除。

## 一句话总结

死代码消除是编译器删除“不会影响最终结果的代码”的优化。

在普通程序中，这是好事。

但在基准测试中，如果被测代码的结果没有被使用，编译器可能把它优化掉，导致测试结果虚假。

所以基准测试必须让被测结果产生可观察影响，或者使用 JMH 的 `Blackhole` 等工具防止被测代码被消除。
