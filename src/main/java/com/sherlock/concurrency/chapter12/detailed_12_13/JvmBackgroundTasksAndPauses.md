# JVM 后台任务、重复运行和显式暂停

## 问题 1

如何理解书中这句话：

> JVM 会使用不同的后台线程来执行辅助任务。当在单次运行中测试多个不相关的计算密集性操作时，一种好的做法是在不同操作的测试之间插入显式的暂停，从而使 JVM 能够与后台任务保持步调一致，同时将被测试任务的干扰降至最低。然而，当测量多个相关操作时，仍旧将相同测试运行多次，如果按照这种方式来排除 JVM 后台任务，那么可能会得出不真实的结果。

## 问题 2

如何理解这句：

> 一种好的做法是在不同操作的测试之间插入显式的暂停。

## 核心意思

这两句话都在讲性能测试中的一个问题：

**你测的不是一个完全干净的世界。**

即使测试代码里只写了一个操作，JVM 和操作系统也可能同时在后台做很多事情。

这些后台活动会影响测试结果。

因此，做性能测试时需要：

```text
重复运行
适当预热
不同测试之间插入暂停
区分哪些开销是外部干扰，哪些开销是被测操作自身的一部分
```

## JVM 后台线程会做什么

JVM 不只是执行你的业务线程。

它还会启动一些后台线程执行辅助任务，例如：

```text
JIT 编译线程
GC 线程
类加载相关线程
引用处理线程
监控线程
信号分发线程
诊断相关线程
```

所以你以为测试过程中只有：

```text
执行被测代码
记录开始时间
记录结束时间
计算耗时
```

但实际可能同时发生：

```text
JIT 正在把热点方法编译成本地机器码
GC 正在回收对象
类正在加载或初始化
操作系统正在调度其他线程
CPU 缓存状态正在变化
```

这些事情都会让某一次测试变慢。

## 为什么要重复运行

一次性能测试结果很可能不可靠。

例如同一个测试运行 5 次：

```text
第 1 次：120ms  可能有类加载和 JIT 预热
第 2 次：90ms
第 3 次：250ms  可能刚好遇到 GC
第 4 次：88ms
第 5 次：89ms
```

如果只跑第 1 次，结果会偏慢。

如果刚好跑到第 3 次，也会误以为代码很差。

所以通常会多跑几次，观察结果是否稳定。

有些情况下，还会取较小值或最小值，用来估计：

```text
当没有明显 JVM 后台干扰时，这段代码大概能跑多快
```

例如：

```text
最小值 = 88ms
```

这个值更接近“少受干扰时的最好表现”。

## 为什么不能总是取最小值

取最小值只适合某些场景。

如果你测的是多个互不相关的计算密集型操作，那么某次测试中遇到 GC、JIT 或系统调度干扰，可以把它看成外部噪声。

例如：

```text
测试 A：纯计算算法 A
测试 B：纯计算算法 B
测试 C：纯计算算法 C
```

这类测试中，某次突然变慢可能只是 JVM 后台任务干扰。

但如果你测的是一组相关操作，情况就不同。

例如：

```text
操作 A 分配大量对象
操作 B 继续处理这些对象
操作 C 触发 GC 或清理
```

这时 GC 不是外部噪声，而是这组操作自然造成的后果。

如果你只取最小值，把包含 GC 的那次结果排除掉，就会低估真实成本。

也就是说：

```text
如果后台开销是被测操作自身引起的，它就是测试结果的一部分。
```

不能简单把它当成噪声排除。

## 什么是显式暂停

显式暂停就是在两个测试之间主动停一会儿。

最简单的形式是：

```java
Thread.sleep(1000);
```

它的目的不是业务逻辑需要，而是测试方法上的隔离。

例如：

```text
测试 ArrayBlockingQueue
暂停 1 秒
测试 LinkedBlockingQueue
暂停 1 秒
测试 SynchronousQueue
```

中间的暂停是为了让上一轮测试的影响尽量消散。

## 为什么不同测试之间要暂停

假设你连续测试两个队列：

```text
先测 ArrayBlockingQueue
再测 LinkedBlockingQueue
```

如果中间不暂停，可能发生：

```text
ArrayBlockingQueue 测试刚结束
        ↓
它引发的 GC、JIT、线程调度余波还没结束
        ↓
马上开始 LinkedBlockingQueue 测试
        ↓
LinkedBlockingQueue 的结果被前一轮污染
```

这时你看到的结果不完全是 `LinkedBlockingQueue` 自己的真实性能。

它混入了上一轮测试遗留下来的影响。

## 显式暂停能降低哪些干扰

显式暂停可以降低这些干扰：

```text
上一轮测试产生的 GC 压力
JIT 编译活动
类加载和初始化
线程池线程创建和退出
操作系统调度波动
CPU 短时间负载波动
前一轮测试对后一轮测试的缓存和内存影响
```

例如：

```text
测试 A 大量创建对象
        ↓
对象变成垃圾，但 GC 还没完全处理
        ↓
马上开始测试 B
        ↓
B 运行过程中刚好触发 GC
        ↓
B 的耗时变长
```

如果中间插入暂停：

```text
测试 A 结束
        ↓
sleep 一段时间
        ↓
GC/JIT/调度活动有机会完成
        ↓
测试 B 开始
        ↓
B 的结果更接近自身表现
```

## 书中 TimedPutTakeTest 的例子

书中 12.13 的驱动程序大致是：

```java
t.test();
Thread.sleep(1000);
t.test();
Thread.sleep(1000);
```

这个 `sleep` 不是功能需求。

它是性能测试中的隔离手段。

作用是让两次测试之间留出缓冲时间，减少前一次测试对下一次测试的影响。

本项目中的 `TimedPutTakeTestDriver` 也保留了这个思想，只是为了缩短本地示例运行时间，把暂停时间调小了。

## 显式暂停不是万能的

显式暂停只能降低干扰，不能完全消除干扰。

因为 JVM 和操作系统的后台活动无法完全由测试代码控制。

例如：

```text
GC 可能仍然在下一轮发生
操作系统可能仍然调度其他进程
CPU 频率可能动态变化
JIT 可能仍然继续编译
```

所以显式暂停只能让测试更稳定一些，不能让测试变成绝对精确。

## 什么时候应该暂停

适合插入显式暂停的场景：

```text
测试多个不相关的操作
比较多个不同实现
上一轮测试可能产生明显 GC 或线程调度余波
希望每组测试之间尽量独立
```

例如：

```text
比较 ArrayBlockingQueue 和 LinkedBlockingQueue
比较 synchronized 和 ReentrantLock
比较不同线程池参数
比较不同缓存实现
```

这些测试之间最好有间隔。

## 什么时候不能简单排除后台任务

如果后台任务是被测操作自然引起的，就不能简单排除。

例如：

```text
某个队列实现因为频繁分配节点，导致更多 GC
某个缓存实现因为对象创建多，导致内存压力更高
某个算法因为产生大量临时对象，导致周期性停顿
```

这些 GC 和停顿就是这个实现的真实成本。

如果你通过取最小值或长时间暂停把它们完全避开，测试结果会过于乐观。

## 一句话总结

JVM 后台任务会干扰性能测试，所以不同测试之间插入显式暂停，可以让上一轮测试的余波尽量消散，减少对下一轮测试的污染。

但如果这些后台开销是被测代码自身造成的，它们就不是噪声，而是真实成本，不能简单排除。
