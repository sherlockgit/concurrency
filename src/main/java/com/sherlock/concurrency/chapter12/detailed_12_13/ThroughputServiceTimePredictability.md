# 吞吐量、服务时间和可预测性

## 问题

如何理解书中这段话：

> 到目前为止，我们的重点是吞吐量的测量，这通常是并发程序最重要的性能指标。但有时候，我们还需要知道某个动作经过多长时间才能执行完成，这时就要测量服务时间的变化情况。而且，如果能获得更小的服务时间变动性，那么更长的平均服务时间是有意义的。“可预测性”同样是一个非常有价值的性能特征。通过测量变动性，使我们能回答一些关于服务质量的问题，例如“操作在 100 毫秒内成功执行的百分比是多少？”

## 核心意思

这段话的核心是：

**并发程序不只要看整体吞吐量，还要看单个操作的响应时间是否稳定。**

前面 12.12 和 12.13 的 `TimedPutTakeTest` 主要测的是吞吐量。

例如：

```text
每秒能完成多少次 put/take
每个元素平均耗费多少纳秒
单位时间内系统能处理多少工作
```

这些指标能说明系统整体处理能力。

但它们不能完整回答：

```text
某一次具体操作要等多久？
大多数操作是否都能很快完成？
是否有少数操作会偶尔卡很久？
用户请求是否能稳定地在 100ms 内返回？
```

这些问题就属于服务时间和可预测性的范围。

## 什么是吞吐量

吞吐量关注的是：

```text
单位时间内完成了多少工作
```

例如：

```text
每秒处理 10000 个请求
每秒完成 50000 次 put/take
每秒写入 200MB 数据
```

吞吐量回答的是：

```text
系统整体处理能力有多强？
```

在并发程序中，吞吐量通常非常重要，因为多线程、线程池、队列、锁优化等很多技术，最终都希望提高系统整体处理能力。

## 什么是服务时间

服务时间关注的是：

```text
单个操作从开始到完成用了多久
```

例如：

```text
一次 put 操作用了多久？
一次 take 操作用了多久？
一次 HTTP 请求多久返回？
一次数据库查询多久完成？
```

服务时间回答的是：

```text
某个具体任务自己要等多久？
```

吞吐量高，不代表每个任务都快。

例如：

```text
系统每秒能处理 10000 个请求
```

这个吞吐量看起来不错。

但如果其中一些请求要等 5 秒才返回，那么用户体验仍然可能很差。

## 平均服务时间的问题

平均值有用，但它会掩盖波动。

例如两个系统：

```text
系统 A：
平均响应时间 100ms
有些请求 10ms
有些请求 1000ms

系统 B：
平均响应时间 130ms
大多数请求都在 120ms 到 140ms 之间
```

如果只看平均值：

```text
系统 A 更快，因为 100ms < 130ms
```

但如果关心稳定性：

```text
系统 B 可能更好，因为响应时间更可预测
```

这就是书中说的：

**如果能获得更小的服务时间变动性(服务时间波动)，那么更长的平均服务时间是有意义的。**

换句话说：

```text
平均慢一点，但每次都比较稳定，有时比平均快但偶尔特别慢更有价值。
```

## 什么是服务时间变动性

服务时间变动性指的是：

```text
每次操作耗时波动有多大
```

低变动性：

```text
第 1 次请求：101ms
第 2 次请求：99ms
第 3 次请求：103ms
第 4 次请求：100ms
```

高变动性：

```text
第 1 次请求：20ms
第 2 次请求：30ms
第 3 次请求：900ms
第 4 次请求：50ms
```

即使两个系统平均值接近，低变动性的系统也更容易满足服务质量要求。

## 什么是可预测性

可预测性就是：

```text
我大致知道一次操作会在多久内完成
不会偶尔突然卡很久
```

对于用户请求、交易系统、支付系统、搜索系统、接口服务来说，可预测性非常重要。

例如：

```text
99% 的请求必须在 100ms 内返回
99.9% 的请求必须在 500ms 内返回
```

这类要求不是单纯的吞吐量指标，而是服务质量指标。

## 什么是“操作在 100 毫秒内成功执行的百分比”

这句话其实是在讲百分位延迟和服务质量。

它不是问：

```text
平均耗时是多少？
```

而是问：

```text
有多少比例的操作能在 100ms 内完成？
```

例如测试结果如下：

```text
总请求数：10000
100ms 内完成：9700
超过 100ms：300
```

那么：

```text
100ms 内成功执行的比例 = 9700 / 10000 = 97%
```

这就可以回答服务质量问题：

```text
97% 的操作能在 100ms 内完成。
```

比只说“平均 80ms”更有信息量。

## 百分位延迟

实际系统中常用百分位延迟描述可预测性。

例如：

```text
P50 = 50% 的请求在这个时间内完成
P90 = 90% 的请求在这个时间内完成
P95 = 95% 的请求在这个时间内完成
P99 = 99% 的请求在这个时间内完成
```

举例：

```text
P50 = 30ms
P95 = 100ms
P99 = 300ms
```

含义是：

```text
50% 的请求在 30ms 内完成
95% 的请求在 100ms 内完成
99% 的请求在 300ms 内完成
```

这样就能看出尾部延迟。

平均值可能很好看，但 P99 很差，说明少量请求会非常慢。

## 吞吐量和服务时间的区别

可以这样区分：

```text
吞吐量：
系统整体每秒能处理多少工作。

服务时间：
某一个具体操作要花多久完成。

服务时间变动性：
不同操作之间耗时差异有多大。

可预测性：
系统是否能稳定地在某个时间范围内完成操作。
```

例如餐厅：

```text
吞吐量：
这家餐厅每小时能接待 200 位客人。

服务时间：
某位客人从点餐到上菜等了 15 分钟。

服务时间变动性：
有些客人 5 分钟上菜，有些客人 40 分钟才上菜。

可预测性：
绝大多数客人都能在 20 分钟内上菜。
```

## 和并发程序的关系

并发程序常常能提高吞吐量，但也可能增加服务时间波动。

原因包括：

```text
线程竞争锁
线程池排队
GC 暂停
CPU 调度延迟
I/O 阻塞
队列积压
缓存失效
上下文切换
```

所以性能测试不能只看：

```text
每秒处理多少请求
```

还要看：

```text
每个请求耗时分布
P95/P99 延迟
超过阈值的请求比例
是否存在偶发长尾
```

## 一句话总结

吞吐量衡量的是系统整体处理能力。

服务时间衡量的是单个操作完成需要多久。

服务时间变动性衡量的是操作耗时是否稳定。

可预测性衡量的是系统能否稳定满足类似“100ms 内完成 95% 请求”这样的服务质量目标。

所以这段话真正想表达的是：

**并发程序不只是要处理得多，还要处理得稳定、可预测。**
