# JIT 自适应优化和性能测试失真

## 问题

如何理解书中这段话：

> 运行时编译器根据收集到的信息对已编译的代码进行优化。JVM 可以与执行过程特定的信息来生成更优的代码，这意味着在编译某个程序的方法 M 时生成的代码，将可能与编译另一个不同程序中的方法 M 时生成的代码不同。在某些情况下，JVM 可能会基于一些只是临时有效的假设进行优化，并在这些假设失效时抛弃已编译的代码。因此，微测试程序不仅要大致判断单线程程序的使用模式，还需要尽量覆盖该应用程序中将执行的代码路径集合。否则，动态编译器可能会针对一个单线程测试程序进行一些专门优化，但只要在真实的应用程序中略微包含一些并行，都会使这些优化不复存在。因此，即使你只是想测试单线程的性能，也应该将单线程的性能测试与多线程的性能测试结合在一起。在 TimedPutTakeTest 中不会出现这个问题，因为即使在最小的测试用例中都使用了两个线程。

## 核心意思

这段话在讲：

**JVM 的 JIT 编译器会根据程序实际运行方式优化代码。如果性能测试的运行方式和真实应用差异太大，测试结果就可能失真。**

Java 程序不是一启动就把所有代码编译成固定不变的机器码。

JVM 通常会经历这样的过程：

```text
先解释执行
        ↓
收集运行时信息
        ↓
发现热点方法
        ↓
JIT 编译成本地机器码
        ↓
根据运行时信息继续优化
        ↓
如果优化假设失效，可能撤销优化或重新编译
```

所以，同一段 Java 源码，在不同运行场景下，最终生成的机器码可能不同。

## 什么是运行时编译器

运行时编译器通常指 JIT 编译器。

JIT 是 Just-In-Time 的缩写，意思是“即时编译”。

它不是在程序启动前一次性完成所有编译，而是在程序运行过程中观察代码执行情况，再把热点代码编译成本地机器码。

JIT 会收集很多运行时信息，例如：

```text
哪个方法被频繁调用
某个接口调用实际总是哪个实现类
某个分支大多数时候走哪边
某个锁是否真的发生竞争
某个对象是否逃出当前线程
某段代码是否只在单线程下执行
```

这些信息会影响 JIT 如何优化代码。

## 为什么同一个方法 M 可能生成不同代码

书里说：

```text
编译某个程序的方法 M 时生成的代码，
可能与编译另一个不同程序中的方法 M 时生成的代码不同。
```

原因是 JIT 不是只看源码，它还看运行时上下文。

例如：

```java
void add(Object value) {
    list.add(value);
}
```

在程序 A 中：

```text
list 永远是 ArrayList
```

JIT 观察到这一点后，可能会大胆优化：

```text
把 List.add 虚方法调用直接变成 ArrayList.add
把 ArrayList.add 内联进来
继续优化数组写入逻辑
```

在程序 B 中：

```text
list 有时是 ArrayList
list 有时是 LinkedList
list 有时是其他 List 实现
```

JIT 就不能做同样激进的优化。

因为调用目标不稳定。

所以同一个源码方法，在不同程序中可能生成不同机器码。

## 什么是基于临时假设的优化

JIT 会根据当前观察到的事实做假设。

例如：

```text
这个方法目前只被一个线程调用
这个接口调用目前只有一个实现类
这个分支目前几乎总是 true
这个锁目前没有发生竞争
这个对象目前没有逃出方法
```

这些假设在当前阶段可能成立，但不一定永远成立。

JIT 可能先基于这些假设生成更快的代码。

如果后来假设失效，JVM 可能会：

```text
撤销这份优化代码
回退到解释执行
重新收集信息
重新编译一份更保守或更适合当前场景的代码
```

这就是书里说的：

```text
在这些假设失效时抛弃已编译的代码
```

## 为什么单线程微测试可能失真

假设你写了一个很小的单线程测试：

```java
for (int i = 0; i < 1000000; i++) {
    queue.put(i);
    queue.take();
}
```

JIT 观察到：

```text
只有一个线程
没有真正的锁竞争
调用路径非常固定
对象生命周期很简单
队列总是刚放入又马上取出
```

于是它可能做出一些非常适合这个小测试的优化。

但真实应用可能是：

```text
多个生产者线程
多个消费者线程
线程池调度
队列时满时空
put 和 take 并发发生
锁竞争
GC 压力
更多代码路径
```

这时，单线程测试中成立的假设在真实应用中不成立。

所以单线程微测试测出来的性能，可能只是“这个小测试的性能”，而不是“真实应用中的性能”。

## 什么是代码路径覆盖不足

书中说微测试程序要尽量覆盖应用程序中会执行的代码路径集合。

意思是：

```text
真实应用会走哪些分支、哪些方法、哪些并发路径，测试也应该尽量覆盖。
```

例如真实应用中的队列可能出现：

```text
队列为空，take 阻塞
队列已满，put 阻塞
多个生产者同时 put
多个消费者同时 take
生产者和消费者同时操作
线程被中断
线程池调度带来的排队
```

如果测试只覆盖：

```text
单线程 put
马上单线程 take
```

那测试就没有覆盖真实应用的关键路径。

JIT 也会基于这个过于简单的路径进行优化。

结果可能不可信。

## 为什么“真实应用中略微包含一些并行”就会让优化失效

并发会改变很多运行时事实。

单线程测试中可能成立：

```text
锁没有竞争
对象没有被多个线程访问
共享变量没有真正共享
内存可见性问题不会暴露
队列不会同时被 put 和 take
```

但只要真实应用中有一点并行，这些事实就可能改变：

```text
锁开始竞争
对象被多个线程访问
共享变量需要内存可见性保证
队列头尾可能同时被操作
线程调度和上下文切换出现
```

于是 JIT 在单线程测试中做的某些专门优化可能不再适用。

## 为什么单线程测试也应该和多线程测试结合

如果你只关心单线程性能，可以做单线程测试。

但如果这段代码最终会在并发应用中使用，就不能只看单线程测试。

更合理的做法是：

```text
单线程测试：看最低成本、基础开销、纯算法效率。
多线程测试：看锁竞争、内存可见性、调度开销、可伸缩性。
真实负载测试：看接近生产环境的整体表现。
```

单线程测试能回答：

```text
这段代码在没有竞争时最快能多快？
```

多线程测试能回答：

```text
这段代码在真实并发访问下还能不能保持吞吐量？
```

两个问题都重要，但不能互相替代。

## 为什么 TimedPutTakeTest 不容易出现这个问题

书中说：

```text
在 TimedPutTakeTest 中不会出现这个问题，
因为即使在最小的测试用例中都使用了两个线程。
```

`TimedPutTakeTest` 的最小配置通常也是：

```text
1 个生产者
1 个消费者
```

也就是说，至少有两个线程同时操作队列：

```text
Producer thread -> put
Consumer thread -> take
```

这比单线程循环：

```text
put
take
put
take
```

更接近真实的生产者-消费者场景。

它至少能覆盖：

```text
两个线程并发访问
队列同步逻辑
线程调度
put/take 的交错执行
锁竞争或条件等待的可能性
```

所以 JIT 不太容易把它优化成一个只适合单线程的假场景。

## 一个简单对比

单线程微测试：

```java
for (int i = 0; i < n; i++) {
    queue.put(i);
    queue.take();
}
```

问题是：

```text
没有真实生产者
没有真实消费者
没有并发 put/take
没有线程调度
没有真正的锁竞争
```

生产者-消费者测试：

```text
Producer thread:
    queue.put(x)

Consumer thread:
    queue.take()
```

更接近真实场景：

```text
put 和 take 可能交错
线程可能竞争
队列可能为空或满
同步机制会真实参与
```

## 一句话总结

JIT 会根据程序实际运行方式优化代码。

如果性能测试太简单、太单线程、太脱离真实应用，JIT 可能生成一份只适合测试场景的机器码，导致测试结果过于乐观。

所以性能测试应该尽量覆盖真实代码路径，并把单线程测试和多线程测试结合起来看。
