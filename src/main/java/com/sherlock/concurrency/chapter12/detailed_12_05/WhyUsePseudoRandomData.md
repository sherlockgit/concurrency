# 为什么测试数据不能太可预测

## 问题

如何理解书中这句话：

```text
要确保测试程序能正确地测试所有要点，
就一定不能让编译器可以预先猜测到校验和的值。
使用连续的整数作为测试数据并不是一种好办法，
因为得到的结果总是相同的，
而一个智能的编译器通常可以预先计算出这个结果。
```

## 12.5 中的测试思路

在 12.5 的 `PutTakeTest` 中，测试逻辑是：

```text
生产者 put 一批数字，并累加 putSum；
消费者 take 一批数字，并累加 takeSum；
最后断言 putSum == takeSum。
```

如果缓冲区正确，那么：

```text
放进去的数字总和 == 取出来的数字总和
```

这可以帮助发现：

```text
元素是否丢失；
元素是否重复；
元素是否被篡改。
```

## 为什么不用连续整数

如果测试数据是连续整数：

```java
for (int i = 0; i < n; i++) {
    buffer.put(i);
    sum += i;
}
```

那么校验和是一个固定公式：

```text
0 + 1 + 2 + ... + (n - 1) = n * (n - 1) / 2
```

也就是说，结果太容易预测。

如果每个线程都放入固定范围的连续整数，那么最终总和也很容易提前计算出来。

## 编译器为什么可能影响测试

现代 JVM 的 JIT 编译器会做很多优化。

如果它发现某段循环的结果是固定的，就可能把一部分计算提前折叠掉。

例如原本代码是：

```java
int sum = 0;
for (int i = 0; i < n; i++) {
    sum += i;
}
```

从数学上看，它等价于：

```java
int sum = n * (n - 1) / 2;
```

在某些场景下，智能编译器可能利用这种规律减少实际计算。

这样会削弱测试的意义。

测试原本想验证的是：

```text
每个元素是否真的生成；
每个元素是否真的 put 到缓冲区；
每个元素是否真的被 take 出来；
消费者拿到的数据是否与生产者放入的数据一致。
```

如果数据和校验和太容易预测，测试就更容易被优化或规律掩盖。

## 为什么使用伪随机数据

书中的测试使用伪随机数：

```java
int seed = ...
for (...) {
    buffer.put(seed);
    sum += seed;
    seed = xorShift(seed);
}
```

这样每个线程的数据序列依赖运行时信息：

```text
线程对象 hashCode；
System.nanoTime；
XorShift 递推结果。
```

编译器很难在编译期提前知道最终校验和。

因此测试更能迫使程序真实执行：

```text
生成数据；
put 数据；
take 数据；
累加校验和；
比较校验和。
```

## 规律数据还可能掩盖错误

连续整数还有另一个问题：太有规律。

如果缓冲区有 bug，导致：

```text
丢失一个元素；
重复另一个元素。
```

某些情况下，总和可能刚好不变。

伪随机数据不能百分百避免这种漏检，但可以降低这种错误被规律数据掩盖的概率。

## 总结

一句话理解：

```text
测试数据要足够不可预测，
让校验和真正来自运行时的数据流，
而不是一个编译器或人都能提前算出的固定公式。
```
