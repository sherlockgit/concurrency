# ReadWriteLock 的实现策略选择

## 问题

将书中关于 `ReadWriteLock` 可选实现策略的几点写下详细例子：

```text
释放优先
读线程插队
重入性
降级
升级
```

## 背景

读写锁不是只有一种固定实现。

它的基本语义很简单：

```text
读锁之间可以共享。
写锁是独占的。
写锁会排斥读锁和写锁。
```

但在读锁和写锁互相竞争时，不同实现可以有不同策略。

例如：

```text
写锁释放后，优先唤醒读线程还是写线程？
有写线程等待时，新来的读线程能不能插队？
读写锁是否支持重入？
持有写锁时能不能再获取读锁？
持有读锁时能不能升级成写锁？
```

这些选择会影响：

```text
吞吐量
公平性
写线程是否饥饿
实现复杂度
死锁风险
```

## 1. 释放优先

释放优先讨论的是：

**当一个写线程释放写锁后，如果等待队列里同时有读线程和写线程，应该优先唤醒谁？**

例如当前状态：

```text
W1 正在持有写锁
R1 正在等待读锁
R2 正在等待读锁
W2 正在等待写锁
```

此时 W1 释放写锁。

读写锁实现需要选择：

```text
方案 A：优先唤醒 R1、R2
方案 B：优先唤醒 W2
方案 C：谁先排队就唤醒谁
```

### 优先唤醒读线程

时间线：

```text
1. W1 持有写锁
2. R1、R2 等待读锁
3. W2 等待写锁
4. W1 释放写锁
5. R1、R2 同时获得读锁
6. W2 继续等待
```

优点：

```text
读线程可以批量放行
读多写少场景下吞吐量高
多个读操作可以并发执行
```

缺点：

```text
如果读线程持续不断到来，写线程可能等待很久
极端情况下可能造成写线程饥饿
```

### 优先唤醒写线程

时间线：

```text
1. W1 持有写锁
2. R1、R2 等待读锁
3. W2 等待写锁
4. W1 释放写锁
5. W2 获得写锁
6. R1、R2 继续等待
```

优点：

```text
写线程不容易饥饿
数据更新能更快发生
适合写操作不能长期延迟的场景
```

缺点：

```text
读线程并发优势可能下降
读吞吐量可能降低
```

### 按请求顺序唤醒

时间线：

```text
1. W1 持有写锁
2. R1 先排队
3. W2 后排队
4. R2 再排队
5. W1 释放写锁
6. R1 先获得读锁
7. 后续根据队列顺序继续处理
```

优点：

```text
公平性更好
不容易让某类线程一直被插队
```

缺点：

```text
可能牺牲吞吐量
读线程不能总是批量最大化放行
```

## 2. 读线程插队

读线程插队讨论的是：

**如果当前锁由读线程持有，但已经有写线程在等待，那么新来的读线程能不能立刻获得读锁？**

例如：

```text
R1 正在持有读锁
W1 正在等待写锁
R2 新到达，想获取读锁
```

这时有两种策略。

### 允许 R2 插队

时间线：

```text
1. R1 持有读锁
2. W1 到达，等待写锁
3. R2 到达
4. R2 直接获得读锁，与 R1 并发读取
5. W1 继续等待
```

优点：

```text
读吞吐量高
读线程延迟低
适合读操作远多于写操作的场景
```

缺点：

```text
如果读线程源源不断到来，W1 可能一直等不到
容易造成写线程饥饿
```

写线程饥饿例子：

```text
R1 持有读锁
W1 等待写锁
R2 插队获得读锁
R3 插队获得读锁
R4 插队获得读锁
...
W1 一直等不到所有读锁释放
```

### 不允许 R2 插队

时间线：

```text
1. R1 持有读锁
2. W1 到达，等待写锁
3. R2 到达
4. 因为 W1 已经在等待，R2 也排队等待
5. R1 释放读锁
6. W1 获得写锁
7. W1 释放写锁后，R2 再获得读锁
```

优点：

```text
写线程不容易饥饿
等待中的写操作能尽快执行
```

缺点：

```text
新来的读线程需要等待
读吞吐量可能下降
```

## 3. 重入性

重入性讨论的是：

**读写锁是否允许同一个线程重复获得自己已经持有的锁。**

例如：

```java
readLock.lock();
try {
    readLock.lock();
    try {
        // 同一个线程第二次获得读锁
    } finally {
        readLock.unlock();
    }
} finally {
    readLock.unlock();
}
```

如果支持重入，这段代码可以正常执行。

如果不支持重入，第二次 `readLock.lock()` 可能会把自己卡住。

### 为什么重入有用

假设有两个方法：

```java
public String get(String key) {
    readLock.lock();
    try {
        return doGet(key);
    } finally {
        readLock.unlock();
    }
}

private String doGet(String key) {
    readLock.lock();
    try {
        return map.get(key);
    } finally {
        readLock.unlock();
    }
}
```

如果读锁不可重入，`get` 已经拿到读锁，再调用 `doGet` 时就会再次请求读锁，可能自己等自己。

可重入锁允许这种调用结构。

`ReentrantReadWriteLock` 从名字上就能看出：它支持重入。

### 重入的代价

支持重入需要维护：

```text
当前线程持有读锁的次数
当前线程持有写锁的次数
释放锁时是否只是减少重入计数
真正释放锁的时机
```

所以实现复杂度更高。

## 4. 降级

降级讨论的是：

**一个线程持有写锁时，能不能在不释放写锁的情况下获取读锁，然后释放写锁，只保留读锁。**

这叫锁降级：

```text
写锁 -> 读锁
```

典型写法：

```java
writeLock.lock();
try {
    // 修改共享数据
    updateData();

    // 在持有写锁时获取读锁
    readLock.lock();
} finally {
    // 释放写锁，但继续持有读锁
    writeLock.unlock();
}

try {
    // 以读锁身份继续读取刚刚更新后的数据
    readData();
} finally {
    readLock.unlock();
}
```

### 为什么需要降级

场景：

```text
缓存失效，需要刷新缓存。
刷新缓存需要写锁。
刷新完成后，还要基于新缓存做一段读取逻辑。
```

如果写完后直接释放写锁，再重新获取读锁：

```java
writeLock.lock();
try {
    updateCache();
} finally {
    writeLock.unlock();
}

readLock.lock();
try {
    readCache();
} finally {
    readLock.unlock();
}
```

中间存在空窗期：

```text
释放写锁
        ↓
还没拿到读锁
        ↓
另一个写线程可能进来修改缓存
        ↓
当前线程读到的可能不是自己刚刚更新后的版本
```

降级可以避免这个空窗期。

### 降级时间线

```text
1. 线程 T 获得写锁
2. T 更新数据
3. T 在仍持有写锁时获得读锁
4. T 释放写锁
5. T 继续持有读锁读取数据
6. T 释放读锁
```

在第 3 到第 4 步之间，T 同时持有写锁和读锁。

释放写锁后，其他读线程可以进来共享读锁，但写线程仍然会被 T 的读锁挡住。

这保证 T 读到的数据不会被其他写线程改掉。

### ReentrantReadWriteLock 支持降级

`ReentrantReadWriteLock` 支持：

```text
持有写锁时，再获取读锁。
```

所以它支持锁降级。

## 5. 升级

升级讨论的是：

**一个线程持有读锁时，能不能不释放读锁，直接获取写锁。**

这叫锁升级：

```text
读锁 -> 写锁
```

看起来很自然：

```java
readLock.lock();
try {
    if (needUpdate()) {
        writeLock.lock();
        try {
            updateData();
        } finally {
            writeLock.unlock();
        }
    }
} finally {
    readLock.unlock();
}
```

但这通常很危险。

## 为什么升级容易死锁

假设两个读线程同时想升级：

```text
R1 持有读锁
R2 持有读锁
```

它们都发现需要写入，于是都尝试获取写锁：

```text
R1 等待写锁
R2 等待写锁
```

但写锁要求：

```text
所有读锁都释放后，写锁才能获得。
```

问题是：

```text
R1 不释放读锁，因为它在等写锁
R2 不释放读锁，因为它也在等写锁
```

于是形成：

```text
R1 等 R2 释放读锁
R2 等 R1 释放读锁
```

这就是死锁。

时间线：

```text
1. R1 获得读锁
2. R2 获得读锁
3. R1 尝试升级为写锁，等待 R2 释放读锁
4. R2 尝试升级为写锁，等待 R1 释放读锁
5. 两者都不释放读锁
6. 死锁
```

## 为什么多数读写锁不支持升级

书中说：

```text
大多数读写锁实现并不支持升级
```

原因就是：

```text
如果没有显式的升级协议，很容易造成死锁。
```

`ReentrantReadWriteLock` 支持降级，但不支持从读锁直接升级为写锁。

如果当前线程持有读锁，再去请求写锁，通常会等待，而这个等待很可能永远无法满足。

## 正确的升级替代方式

如果读的时候发现需要写，常见做法是：

```text
先释放读锁
再获取写锁
获取写锁后重新检查条件
```

示例：

```java
readLock.lock();
try {
    if (!needUpdate()) {
        return data;
    }
} finally {
    readLock.unlock();
}

writeLock.lock();
try {
    // 必须重新检查，因为释放读锁到获得写锁之间，状态可能已经被别人改了
    if (needUpdate()) {
        updateData();
    }
    return data;
} finally {
    writeLock.unlock();
}
```

重点是：

```text
释放读锁后，状态可能发生变化。
所以获得写锁后必须重新检查条件。
```

## 五个策略的总结

```text
释放优先：
写锁释放后，优先唤醒读线程、写线程，还是按队列顺序唤醒。

读线程插队：
有写线程等待时，新来的读线程能不能直接获得读锁。

重入性：
同一个线程能不能重复获得自己已经持有的读锁或写锁。

降级：
持有写锁时，能不能再获取读锁，然后释放写锁，只保留读锁。

升级：
持有读锁时，能不能不释放读锁就获取写锁。
```

## 一句话总结

读写锁的核心不是只有“读读共享、读写互斥”这么简单。

真正复杂的是竞争策略：

```text
谁先被唤醒
读线程能不能插队
写线程会不会饥饿
锁能不能重入
能不能降级
能不能升级
```

这些实现选择决定了读写锁在不同场景下的公平性、吞吐量、死锁风险和活跃性表现。
