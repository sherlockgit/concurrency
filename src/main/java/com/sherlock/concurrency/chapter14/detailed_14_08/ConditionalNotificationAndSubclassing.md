# 条件通知、单次通知与子类化

## 问题

如何理解这句话：

```text
在使用条件通知或单次通知时，一些约束条件值得子类化过程变得更加复杂。
要想支持子类化，那么在设计类时需要保证：如果在实施子类化时违背了条件通知或单次通知的某个需求，
那么在子类中可以增加合适的通知机制来代表基类。
```

## 简单理解

这句话的核心意思是：

```text
条件通知和单次通知都是优化，但它们依赖一些很强的前提。
如果这个类允许被继承，子类可能会改变这些前提，从而导致通知不够，线程永远等不到。
```

换句话说：

```text
通知策略也是类的同步协议的一部分。
子类不是简单增加几个方法，它可能改变整个等待和通知关系。
```

## 结合 14-8 的条件通知理解

`14-8` 中的条件通知写法类似这样：

```java
public synchronized void put(V value) throws InterruptedException {
    while (isFull()) {
        wait();
    }

    boolean wasEmpty = isEmpty();
    doInsert(value);

    if (wasEmpty) {
        notifyAll();
    }
}
```

它的逻辑是：

```text
只有缓冲区从“空”变成“非空”时，才通知等待线程。
```

这个优化成立的前提是：

```text
put 只需要唤醒等待“非空”的消费者。
```

也就是说，基类认为：

```text
我只关心一个条件变化：empty -> not empty。
```

## 子类可能破坏这个前提

假设子类增加了一个新方法：

```java
public synchronized void awaitHalfFull() throws InterruptedException {
    while (size() < capacity() / 2) {
        wait();
    }
}
```

这个方法等待的条件是：

```text
缓冲区至少达到一半容量。
```

现在假设容量是 4：

```text
当前已有 1 个元素。
某个线程正在 awaitHalfFull() 中等待。
另一个线程执行 put，元素数量从 1 变成 2。
```

此时：

```text
awaitHalfFull 的条件已经满足。
```

但是对 14-8 的基类 `put` 来说：

```text
wasEmpty == false
```

所以基类不会调用：

```java
notifyAll();
```

结果是：

```text
等待 awaitHalfFull 的线程可能永远睡着。
```

这就是子类化带来的问题：子类增加了新的条件谓词，但基类的通知策略并不知道这个新条件。

## 单次通知 notify 也有类似问题

`notify()` 只随机唤醒一个等待线程。

它通常要求：

```text
所有等待线程等待的是同一种条件。
唤醒任意一个等待线程都能推进程序。
```

如果子类增加了新的等待条件，就可能出现这种情况：

```text
基类调用 notify。
notify 随机唤醒了一个等待条件仍然不满足的线程。
真正条件已经满足的线程没有被唤醒。
程序可能卡住。
```

所以，`notify()` 比 `notifyAll()` 更依赖严格前提。

一旦允许子类扩展新的状态依赖方法，单次通知的正确性就更难保证。

## 为什么说子类化过程变复杂

普通方法的子类化通常只需要考虑：

```text
子类是否保持方法语义？
子类是否破坏字段不变性？
```

但带有条件队列的类，还必须额外考虑：

```text
子类是否增加了新的条件谓词？
子类是否增加了新的 wait？
基类的 notify/notifyAll 是否足以唤醒子类新增的等待线程？
子类是否需要在状态变化后补充通知？
```

因此，子类化不再只是“扩展功能”，而是要参与整个同步协议设计。

## 如果要支持子类化，应该怎么设计

如果一个使用条件通知或 `notify()` 的类想支持继承，通常要做下面几件事之一。

## 方案一：不要过度优化，直接使用 notifyAll

最保守的做法是：

```java
notifyAll();
```

而不是：

```java
if (wasEmpty) {
    notifyAll();
}
```

这样即使子类增加了新的等待条件，只要状态发生变化，等待线程都有机会醒来重新检查。

缺点是性能上可能多唤醒一些线程。

优点是正确性更容易保证。

## 方案二：提供 protected 通知钩子

基类可以把状态变化后的通知逻辑抽出来：

```java
protected void notifyStateChanged() {
    notifyAll();
}
```

然后子类可以覆盖这个方法，增加自己的通知规则。

这样子类如果增加了新的条件谓词，也能补充通知机制。

## 方案三：明确禁止子类增加新的等待条件

如果类的设计文档中明确规定：

```text
子类不能增加新的 wait 条件。
子类不能改变条件谓词。
子类不能覆盖关键同步方法。
```

那么基类可以继续使用条件通知优化。

但是这种约束必须写清楚，否则子类作者很容易无意破坏同步协议。

## 方案四：把类或关键方法声明为 final

如果不想支持子类参与同步协议设计，可以直接限制继承：

```java
public final class BoundedBuffer {
    ...
}
```

或者限制关键方法：

```java
public final synchronized void put(V value) {
    ...
}
```

这可以防止子类破坏条件通知的前提。

## 一句话总结

```text
条件通知和 notify 都是建立在“我知道有哪些线程在等什么条件”这个前提上的。
一旦允许子类加入新的等待条件，基类就可能不知道该通知谁，所以继承会变复杂。
```

如果要让这种类支持子类化，就必须让子类有机会补充通知机制，或者明确限制子类不能改变等待条件。
